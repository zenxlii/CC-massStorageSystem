~~~~Recipe Format (table):
Index 1 stores a table containing the
recipe's outputs or results.
Each output should be a sub-table which
contains the item's encoded name with
no shorthands allowed, followed by how
many of said item are produced by the
recipe.
If no number is supplied, it is implied
to be 1.

Index 2 stores a table containing the
recipe's inputs or ingredients.
Each input should be a sub-table which
contains the item's encoded name with
no shorthands allowed, followed by how
many of said item are produced by the
recipe.
The inputs need to be indexed by either
the slot number of the target inventory
they are intended for, or the inventory
reference ID for the crafting type,
depending on the crafting type.
(craftingTable is slot-based).
If no number is supplied, it is implied
to be 1.

Index 3 is the maximum batch size for
this recipe.
Setting this to 1 is safe for any valid
recipe, but you should never set this
higher than any of the amount of output
items that the output container can
store (for slot-based crafting types),
or the maximum stack size for a given
ingredient divided by how much of it is
needed per craft, whichever is lower.

Index 4 is the crafting type for this
recipe.
This is used to determine where the
ingredients are sent, where to expect
the results to come from (for slot-
based crafting types), and if the
server should use a slot-based or
inventory-based handler for this
particular recipe.

Index 5 is the priority level assigned
to this recipe and if it is to
override another recipe. Leave this
blank if you don't want to override
anything.

Index 6 lists item encoded names that
should not have this recipe put under
them in the recipeMap.
Good for recipes that return empty
fluid containers.

~~~~Crafting Type Format (table):
These are indexed by name.

Index 1 is one of the following
strings:
>"i"
-Inventory-based crafting type, uses a
different inventory for each input slot
of the final crafting machine and
expects the results to be pulled back
into the system via, say, an import
task.
>"c"
-Slot-based crafting type that is
specific to crafting table recipes.
Makes use of the server turtle's crafty
capabilities.
Use crafting table grid slot numberings
for this (see below).
 1 | 2 | 3 
―――+―――+―――
 4 | 5 | 6 
―――+―――+―――
 7 | 8 | 9 

Index 2 is a table of inventories that
this crafting type inserts items into.
If a crafting type only needs one
inventory, it still must be in a table.
The order of these may be important!
Left as "server" when index 1 is "c".

Index 3 is used by slot-based crafting
types to specify which slot(-s) that
ingredients are to be placed in.

Index 4 is used by "s"-class crafting
types to specify which slots may have
output items for post-operation
extraction.

~~~~Creating New Recipes:
It is recommended that you use a tool
like newRecipeGenerator.lua to create
crafting recipes for massStorageSystem,
due to the data density of the format.
New recipe types, however, will need to
be implemented as functions in code for
the time being.

~~~~Resource Pools:
A number of common resources (such as
Iron or Redstone) have multiple item
forms with differing levels of material
density (e.g. 1 Iron Block = 9 Iron
Ingots = 81 Iron Nuggets), and these
can be defined as a resource pool that
can be drawn from and added to as if
all of these forms were the same item,
while MSS condenses them into higher-
density forms in the background and
decondenses them as needed to satisfy
requests and the likes.

~~~~Resource Pool Format (table):
The master table is indexed by user-
supplied names for the to-be-pooled
resources.

Each pool's table is formatted as
follows:

Index 1 is an array of arrays that
contain:
>The encoded name of the to-be-pooled
item.
>The value of this item in terms of
raw pool resource units (usually this
will be the number of least-dense-form
item copies this particular item is
equivalent to).

Index 2 is an array of recipe
definition pairs which link two items
in the pool together. You will need to
ensure that every item in the pool can
reach every other item in the pool when
factoring in the other recipes listed
in here (e.g. Iron Block <> Iron Ingot
and Iron Ingot <> Iron Nugget would be
sufficient for a pool of Iron Block,
Iron Ingot and Iron Nugget).
